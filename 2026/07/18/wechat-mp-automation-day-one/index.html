公众号自动化第一天，AI 把能踩的坑全踩了一遍

发表于2026-07-18|更新于2026-07-19|技术
|浏览量:

我的公众号 bio-ai4s 今天正式开张了。定位很垂直，只发生物 AI4S 相关的内容：算法、模型、数据集、AI Scientist 这类。

运营思路也很 2026：写稿、排版、生成封面、推送草稿，全部交给 AI（就是我的 nanobot）自动完成，我只负责在手机上的公众号助手 App 里点一下“发表”。

理想很丰满。现实是，第一天，AI 把微信生态能踩的坑全部踩了一遍。一个都没落下。

管线长什么样

先说说设计，其实很简单：

  1. 我给 AI 一个题材（比如今天这篇，是 CZ Biohub 在 Kaggle 上办的细胞追踪比赛）
  2. AI 扒资料、写稿、把 Markdown 排成微信能吃的 HTML
  3. 顺手用 AI 画一张封面图
  4. 通过 md2wechat 调微信官方 API，直接推送到公众号草稿箱
  5. 我在手机上检查，点发表

个人订阅号没有发表接口（微信 2025 年把 freepublish 下了），所以最后一点还得是人。但除了点那一下，其他全自动。

踩坑实录

下面进入正片。按出场顺序排列。

坑一：微信看到的 IP 和我以为的 IP 不是一个

微信 API 要配 IP 白名单。AI 查了本机公网 IP 给我，我填了，调接口，报错：invalid ip 114.86.72.241, not in whitelist。

等等，114 开头这个是谁？一查，直连出口是 36 开头那个，但访问微信 API 时被运营商分流到了另一条线路。两个 IP 都加进白名单才通。

教训：报白名单错误时，别猜，错误消息里写的那个 IP 才是微信真正看到的你。

坑二：标题叫“AI生成测试文章”

第一篇文章推进草稿箱，我打开一看，标题：《AI生成测试文章》。

md2wechat 的 test-draft 命令自带默认元数据，标题、摘要全是占位文案。摘要更绝：“这是AI生成的微信公众号文章测试”。好家伙，直接自爆。

坑三：整个草稿变成 \u 乱码

AI 调 draft/update 接口改标题，改完我一看，标题变成了 CZ Biohub \u53d1\u4e86\u4e2a\u6bd4\u8d5b...，正文也全灭了，满屏 \u 加十六进制。

原因：Python 的 json.dumps 默认把中文转义成 \uXXXX，而微信服务端不解析这种转义，原样当文本存了。必须用 ensure_ascii=False 把中文按原始 UTF-8 发过去。

这坑最骚的地方在于：它是修别的 bug 时引入的。修一个，炸一片。

坑四：微米符号 μm 变成了 ?m

正文里有几处分辨率，比如 1.625 µm。微信编辑器会把 µ（U+00B5，微符号）直接存成问号。

修复方案：换成希腊字母 μ（U+03BC）。长得一模一样，微信认。

坑五：列表符号集体变性

我在手机上检查排版，发现无序列表的小圆点变成了实心小方块 ▪。没人让它变，编辑器自己动的手。最后干脆把 HTML 里的列表符号显式写成 ▪，你要变就变，变完还是我写的样子。

坑六：参考文献居中

参考文献和链接那几行，被排成了居中对齐。运营文里参考文献居中，视觉效果相当魔幻。规矩立下：参考区块左对齐、小字号、样式朴素。

结局是好的

坑踩完，文章也发出去了。今天是双发：博客一篇，公众号一篇，同一个稿子。

更重要的是，每个坑在踩完之后都被当场写进了 AI 的 skill 文件。现在的避雷清单包括：白名单要看错误消息里的 IP、推送后逐项检查标题摘要作者、JSON 必须 UTF-8 原文、µ 要写成希腊 μ、列表符号显式声明、参考文献左对齐。

也就是说，这些坑它这辈子只会踩一次。下一篇文章开始，就是纯享受成果的阶段了。

一点感想

以前觉得“AI 帮我干活”的意思是 AI 不会犯错。现在理解了，真正的意思是：AI 会犯错，但它把每个错都记下来，然后真的不再犯。

人做不到这一点。人会同一个坑踩三次，然后怪路不平。

今日管线全家福：nanobot（写稿+排版+封面）→ md2wechat（推草稿）→ 微信草稿箱 → 我（点发表）。成本：封面图约 3 毛钱。
