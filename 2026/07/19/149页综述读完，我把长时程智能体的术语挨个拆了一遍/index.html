长时程智能体（一）：149 页综述，先把术语挨个拆清楚

发表于2026-07-19|更新于2026-07-19|读书笔记
|浏览量:

最近 agent 相关的文章里，有几个词出现的频率越来越高：长时程、长运行、自主智能体、自进化、harness。大部分人（包括之前的我）用得很随意，长时程约等于“跑得久”，harness 约等于“脚手架”，具体边界在哪，说不清。

前段时间读了人大 RUC 团队牵头挂出来的综述《Towards Long-Horizon Agents》，149 页，引了差不多 900 篇。这篇不打算复述全文，只拆它最有价值的一块：taxonomy，也就是这套术语体系本身。它把几个天天见但从没人认真区分的概念，挨个给了定义和判定标准。

这个系列一共四篇：术语拆解（本篇）、资源清单、模型侧优化、应用与前沿。

地基：Agent = π_θ ⊕ H

整篇综述建立在一个公式上：

Agent = π_θ ⊕ H

π_θ 是模型权重里学到的东西。H 是 harness，指权重之外、运行时（runtime）包着模型的所有东西：提示词怎么组织、上下文怎么裁剪、记忆读写、工具调用、多 agent 编排、权限钩子、结果验证，全算。

这个公式想说的是：模型不换，只换 harness，你面对的就是另一个 agent。表现、上限、失败方式都会变。所以讨论“某某 agent 行不行”，得先说清楚讨论的是哪一层。

主词：长时程，和四个容易混的近义词

综述干的最重要的一件事，是把“长时程”（long-horizon）和四个近义词切开。先上表：

术语 看什么 内涵 一个例子
长时程 long-horizon 步骤间的依赖结构（dependency structure） 后面的决策依赖前面的结果，错了会累积，有的还不可逆 30 分钟的复杂 bug 修复：定位、改码、跑测试、再改
长运行 long-running 墙钟时长 就是跑得久，步骤可以彼此无关 对一百万个文件重复同一操作，跑三天
自主 autonomous 人在不在 loop 里 治理属性（governance）：多少决策不用人批 每步都要人点头的 agent，也可以是长时程的
自进化 self-evolving 经验留不留得下 经验能不能跨 session 沉淀下来 跑完就忘的是一次性 agent；记住教训的才算自进化
长上下文 long-context 一次调用看多少 token 模型的窗口属性，是个 enabler（前提条件） 1M 窗口的模型，照样可以只做单轮问答

逐个说两句。

长时程的定义是五个词：多步骤、有依赖、会漂移、部分不可逆、经验要复用。注意里面没有“时间长”。一个任务算不算长时程，看它的依赖图（dependency graph）长什么样，不看跑多久。

长运行是最常被混进来的。综述的态度很明确：时长既不是充分条件也不是必要条件。一个 agent 跑了两周，干的活可以全是互不相关的短任务拼接，长时程能力一点没用到。

自主是另一个维度。它回答“谁拍板”，长时程回答“系统能不能在长链条上维持状态”。两个维度自由组合：低自主长时程（比如每步审批的编码 agent）和高自主短任务（比如自动回邮件的脚本）都存在。

自进化的判定标准只有一条：经验有没有被写进持久载体（persistent storage，比如记忆文件、技能库、甚至权重），下次启动还能用上。只在当前任务里反思、改错，任务结束就清零，那叫任务内自适应（within-task adaptation），不算自进化。

长上下文最容易理解，就是个窗口属性。窗口大了，H1 级任务（下面讲）确实能直接被“吞掉”，但它解决不了轨迹超出窗口之后的事。

任务分三级：H1、H2、H3

综述给任务本身分了级，标准只有一条：状态超出上下文窗口的程度。

级别 一句话 特征 典型任务
H1 窗口内（intra-context，分钟级） 轨迹能装进窗口 状态膨胀还能管，失败多是局部错误 多跳问答、几十步的网页操作
H2 超窗口（cross-context，小时到天） 装不下了，但目标本身不变 状态必须压缩、写入外部记忆、做 checkpoint，错一步会连锁 仓库级代码迁移、跑几十轮的深度调研
H3 开放世界（cross-task / open-ended，lifelong 级） 连目标都在变 环境会动，目标会演化，agent 还得跟人和其他 agent 协商 运营一个项目、做科研、跨团队协作

这张表读起来像升级路线，但综述强调它更像“问题性质的分界”。H1 的难题靠工程优化能磨掉，H2 开始必须动结构（记忆、压缩、验证器 verifier），H3 的难点已经不在执行，在于“目标本身是什么”都在变化，得边做边重新对齐。

能力也分三级：C1、C2、C3

任务分级之外，agent 的能力（competence）另有一套分级，看的是它能不能在长跨度上维持三样东西：

级别 名字 维持什么 失败长啥样
C1 连贯 coherence 逻辑不断线 长链条推理不自相矛盾，不丢线索 推到一半忘了前面的约束，开始胡来
C2 对齐 alignment 目标不漂移 当前行为始终服务原始目标，不被中间反馈带偏 指标在涨，但已经不在解决原来的问题
C3 经验性（experientiality） 经验能复用 把跨 session 的经验更新进行为里 同一个坑掉第二次，教训留不住

这套分级的用处在于定位失败。一个 agent 翻车，先问是哪一层断了：推理链断了是 C1，目标跑偏是 C2，经验没留下是 C3。三层对应的修法完全不一样。

从 prompt 到 harness：控制面的三次搬家

术语拆开之后，综述用“控制面（control plane）在哪”串了 agent 技术的五年演进：

阶段 年份 控制面 代表技术 为什么到头了
Prompt Engineering 2020–2023 语言 CoT、ToT、ReAct 措辞控制不了模型没见过的东西：实时信息、私有数据、代码库
Context Engineering 2023–2025 信息 RAG、工具调用、MemGPT 还是开放循环（open-loop），管的是“行动前看到什么”，管不了“行动之后发生什么”
Runtime Harness 2025 至今 整条轨迹 Claude Code、OpenHands、MCP 还没到头，但开始暴露归因（attribution）和迁移性问题

三阶段是嵌套关系，不是替代。今天的 agent 依然在做 prompt 和 context engineering，只是重心移到了轨迹这一层。注意第三阶段的名字：控制对象不再是喂给模型的内容，而是围绕模型的整个运行时。

Harness 里到底装了什么

既然 harness 是主角，综述把它拆成六个组件：

组件 干什么 代表
Execution Loop 多轮交互的主循环 ReAct、CodeAct、Reflexion
Context & Memory 把膨胀的轨迹变成有界工作区 AGENTS.md、Mem0、Voyager 技能库
Tool Interface 接工具、接协议 MCP、A2A、各类 API 适配
Multi-agent 角色分工和编排 LangGraph、AutoGen、Magentic-One
Control & Permission 审批、沙箱、预算、回滚 权限模式、checkpoint 机制
Verification 验证器和评估 PRM、tool-grounded verifier

这个拆法挺实用。看任何一个 agent 系统，拿这六个格子去套，哪格是空的、哪格是堆料，一眼能看出来。

模型侧：七件事

模型内部怎么变强，综述也给了张清单：memory consolidation、context internalization、tool & action learning、RL post-training、verifier-guided optimization、multi-agent co-training、training self-evolution。大体是从“把运行时经验蒸馏回权重”到“在线自进化（online self-evolution）”的一个光谱，细节展开又是一篇文章的量，这里先记着名字。

这套 taxonomy 到底有什么用

我的体会是提供了坐标系。以后读 agent 论文或者看别人吹产品，可以问几个现成的问题：这任务是 H 几？报的提升是改了 π 还是改了 H？失败后的教训写进持久载体了吗？没写进去就别说自进化。跑了两周很唬人，但依赖结构是长链条还是短任务拼接？

几个问题问下来，大部分营销话术自己就散了。

也要说句公道话，这套体系不是没毛病。H 的分级会随着模型窗口扩大而漂移，今天的 H2 明天可能变 H1，“长时程”本身是个移动靶。还有 harness 这个筐实在太大，从 system prompt 到沙箱全算进去，做归因分析的时候并不好用。但有一说一，在 2026 年这个满屏都是 agent 的时间点，有套能落地的术语，比没有强太多。
