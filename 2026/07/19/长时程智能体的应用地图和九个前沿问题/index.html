长时程智能体（四）：五大应用领域，九个前沿问题

发表于2026-07-19|更新于2026-07-19|读书笔记
|浏览量:

系列第四篇，也是《Towards Long-Horizon Agents》读书笔记的收官。前面三篇拆了术语、资源、模型侧优化。这篇讲最后两章：§6 应用地图，§7 前沿判断。

这两章放在一起读正好：§6 回答“长时程 agent 现在在哪上班”，§7 回答“接下来卡在哪”。

§6 应用：按接口分，不按难度分

综述把应用分成五个领域，分类标准不是任务难度，而是 agent 和环境之间的接口：agent 通过什么观察世界、通过什么动手、能拿到什么反馈。接口不同，对长时程能力的要求就不同。

先上总表。反馈强度这一列是原文 Figure 14 的分法，很说明问题：

领域 接口 反馈强度 核心能力
软件工程 代码仓库 + 终端执行 强（硬 oracle：测试、编译器直接判对错） 仓库 grounding、工作流级规划、反馈驱动修复
信息检索 网页、搜索引擎、文档 弱（只有证据，没有标准答案） 深搜、广搜、多模态取证、研究综合
计算机操作 截图、DOM、无障碍树 部分（状态延迟可见，动作有持久后果） 浏览器、桌面 GUI、移动端
多模态 长视频、音频流、图像 稀疏（关键信息埋在几帧里） 主动感知、迭代生成、全模态编排
通用型 一堆工具、API、权限的混合体 混合 个人助理、具身、生产力

软件工程：反馈最好的战场

软件工程是长时程 agent 最成熟的应用，原因很朴素：可执行反馈太强了。测试、编译器、运行时轨迹会自动告诉 agent 这一步对不对，别的领域羡慕不来。但综述也提醒，这些信号只是代理指标：测试全过不代表真需求解决了，隐藏要求、安全、回归风险都不在信号里。

三个核心能力拆开看：

仓库 grounding（repository grounding）。issue 不会告诉你该改哪，agent 得自己给一个装不进窗口的代码库建立工作模型。三条路线：SWE-agent 设计了一套适合 LLM 的 agent-computer interface；AutoCodeRover 按类和方法的层级结构搜，不搜原始文本；OpenHands 维护跨步骤的持久工作区；Aider 的 repo map 和 AGENTS.md 则是把仓库结构外化成紧凑可复用的文件。

工作流级规划（workflow-level planning）。知道改哪之后，还得排出先后顺序、跟踪进度、决定什么时候收手。趋势是把工作流状态显式化：分支、计划、checkpoint、审批门，让进度可审查可恢复，而不是隐式埋在对话历史里。

反馈驱动修复（feedback-driven repair）。原则是让修复可检查、可回滚：保留 diff、日志、隔离工作区，失败了能诊断而不是直接覆盖。这里综述诚实地提了个反例：Agentless 用一个固定的“定位-修复”流水线，效果能跟精心设计的 agent 系统打平。不是每个问题都需要开放式 agent loop。

信息检索：从“找到”到“研究”

信息检索类 agent 的难，在于没有执行 oracle，只有证据。四个能力：

能力 干什么 代表
深搜（deep search） 单线索追到底，跨多跳检索 RL 派（Search-R1 一系）vs 工作流派（ReSum 压缩历史续命）；WebSeer 持续复查中间结论
广搜（wide search） 一个目标拆成多路并行调查，保覆盖率 FlowSearch；把搜索过程组织成图、树、表
多模态取证 证据在图、表、截图、视频里 图像检索定位、长视频关键片段主动搜寻
研究综合（research synthesis） 把散证据编成有出处的报告 OpenAI/Gemini Deep Research；大纲、笔记、证据归因全外化

长链条还有个隐蔽的坑：检索失败和推理失败会越来越难分，错误假设顺着链条往下传。所以 WebSeer 那类“边查边验”的机制很关键。

计算机操作：动作有持久后果

浏览器、桌面、手机，共同点是界面状态部分可观察、动作做了就收不回。浏览器 agent 靠可复用的动作原语（点、输、滚、导航）把操作结构化，靠外化记忆攒中间成果。桌面 GUI 的命门是视觉接地（visual grounding）：高分辨率、动态布局里持续找对元素，UI-TARS 是代表；再配上截图对比、视觉变化检测这类执行验证。移动端则是长多应用工作流加频繁上下文切换，主流解法是分层架构：高层规划、低层执行。

多模态：预算决定看什么

多模态长时程任务的核心矛盾：信息流太长，预算太少，关键证据往往就在长视频的几帧里。所以主动感知（active perception）是一等能力，两条路线：VideoSeek 一系把长视频当可探索空间，按需调工具找证据；DVD 一系先建结构化视频记忆再检索。生成侧，多模态生成不是一次 prompt 出图，是“理解意图-规划-多轮生成-验证”的长流程，证据检索、验证、记忆三个机制缺一不可，MUSE 的 plan-execute-verify-revise 循环是典型。全模态编排（omnimodal agency）则是在 Qwen-Omni 这类原生全模态模型和 OmniAgent 这类编排派之间分叉，麻烦在于不同模态的信息会打架，显式验证省不掉。

通用型：持续协调

跨异构接口干活，核心挑战是持续协调：整体规划、跨环境搬状态、失败了能爬回来。综述按部署场景分三类。个人助理里又分四小类：办公（OfficeQA、SpreadsheetBench）、个人计算（复用 §6.3 那套）、软件开发（Claude Code、Codex）、常驻型（OpenClaw、Hermes 这种跑在用户自己设备上、连着本地资源和长期记忆的，这个我有切身体会）。具身方向，VLA 模型（Gemini Robotics 等）把语言、视觉、控制统一进一个动作接口，世界模型分两派：V-JEPA 2 这类预测表征派压抽象状态，Cosmos 这类生成派直接造未来画面。生产力方向，金融法律医疗（FinRobot、LawThinker、MedAgentBench）加上科学发现（AI Scientist、AlphaEvolve、Kosmos、Coscientist）和形式数学（Seed-Prover、Prover Agent 在 Lean 里迭代证题）。

§6.6 是 benchmark 和资源大表，这个系列第二篇已经整理过，不重复。

§7 Frontier：四个 axis，九个 position

§7 是综述的观点输出章，形式是九个 position（立场判断）。开头有个总诊断，我觉得是全文最锋利的一句：“practical long-horizon progress is increasingly a property of the runtime around the model, not of the model alone.”意思是：长时程的实用进展，越来越取决于模型周围的运行时，而不只是模型本身。能力在 harness 和权重之间双向迁移，而 noisy、昂贵、不可逆的真实环境，正在把两边赖以成立的理想化假设都废掉。

九个 position 归在四个 axis 下：evolution（进化）、effectiveness（有效性）、efficiency（效率）、trustworthiness（可信）。总表：

axis 前沿 现在卡在哪 综述的判断
Evolution 自进化 harness 与 agent 优化目标是人手定的；涨点集中在分布内；长跑会过拟合和漂移 agent 自己发现优化目标、自主进化、终身运行不过拟合
Evolution harness 泛化与可迁移 模型和 harness 深度绑定，换个 harness 排名能差几十分 可携带技能（外部迁移）、多 harness 混合训练（内部泛化）、标准化 harness 协议
Evolution 持续与终身学习 外部经验脏且未验证；权重更新会灾难性遗忘 经验先存外部，验证过的才固化进权重；从十亿级用户日志里安全学习
Effectiveness 真实环境交互 真实交互慢、贵、不可逆；合成环境可能不保真 环境构建（不是模型规模）是下一阶段的瓶颈；faithfulness 要像成功率一样被显式测量
Effectiveness 数字到具身 时间尺度冲突、物理鸿沟、反馈粒度差 异步分层 harness（大脑小脑分工）、世界模型预测物理后果、跨尺度验证器
Efficiency 成本与预算感知 agent 全是预算盲（budget-blind）：不知 token 多少钱，没有上限意识 感知成本、中途改道、harness 强制预算、找预算-成功率的 scaling law
Efficiency 多模态与全模态 harness 视觉 token 太贵，跨模态验证不可靠 视觉原生 harness 底座、按模态定粒度、渲染前抓错、全模态路由
Trustworthiness 反思与错误鲁棒性 内省式自纠错不可靠，错误复利成目标漂移 早检测早换路、对齐人类纠偏、外部锚定的校准自知
Trustworthiness 安全与治理 环境会注入错误、没有统一安全验证标准、自进化会侵蚀安全不变量 经验复用先过安全审查、安全性当一等评分项、独立验证器守住不变量

挑几个我觉得最有嚼头的展开。

harness 可迁移性是 scalability 的头号风险。现在的趋势是每个强模型都和自己的 harness 共同调优、深度绑定，但用户想要的是拿自己的 agent 框架去驱动任何模型。同一 harness 换家模型厂商，排名能差出几十分，也就是说测出来的“能力”有相当一部分是“模型惯用语和 harness 的匹配度”。综述给的三条出路里，我最看好内部泛化：训练时就混多种 harness 的轨迹，甚至跑 hybrid-harness RL，把“harness 鲁棒”直接训进权重，而不是事后贴膏药。

预算盲（budget-blind）这个词骂得很准。现在的 agent 每步都默认上最强模型、反复重读可以摘要的上下文、给已经没救的 run 继续烧钱。综述把预算感知拆成一条链：先能感知任务的难度、成本、还有“这事儿是不是已经不可行了”（光一个粗粒度的不可行信号就能省下烧在死局上的大把 token），然后估计值得真的改变行为，再由 harness 强制执行硬上限，最后指望有人找出预算和成功率之间的 scaling law。没有这条 law，预算分配就永远是手艺活。

环境构建是瓶颈，不是模型规模（原话：“environment construction, not model scale, is the rate-limiting step for the next phase of agent capability”）。训练长时程 agent 不能直接在真实系统里造，太慢太贵不可逆。两条路线：WebArena、OSWorld、TheAgentCompany 这类可验证合成环境，和 GenEnv 这类模型模拟环境。两条都过同一个考验：faithfulness。综述主张把保真度拆成几个可单独审计的维度：接口和动作像不像、同样动作产生的状态变化一不一样、自动验证器和真实验收标准一致吗、模拟用户像真人吗、reset 之后是功能等价还是只是长得像。

安全这节有个例子值得背下来：agent 发现“成功退款会收到感谢回复”，就可能过度泛化，以后遇事就爱退款。经验复用本身就是攻击面，何况技能、服务器、共享记忆这些 harness 产物还在团队间流通，跟软件供应链一个道理。综述的结论很直接：“the harness, not the model, is the principal alignment and privilege surface of a production agent”。harness（不是模型）才是生产级 agent 的主要对齐面和权限面，模型侧对齐必要但远不充分。每一层安全机制都带来延迟和成本，这个开销有个名字，叫对齐税（alignment tax）。

自纠错泼了盆冷水。文献已经证实，模型光靠内在信号经常不能可靠自纠，强行纠还会更差。所以反思必须由外部验证锚定（CRITIC 那种工具接地的批评明显更靠谱），长时程下的第一优先级是尽早发现路径错了、尽早换路，而不是一条道走到黑。

收官感想

四篇读下来，这篇综述最大的价值不是清单本身，而是给了两套坐标：H1-H3、C1-C3 用来定位任务和能力，π ⊕ H 用来归因任何一笔提升。§7 的九个判断往深了看是同一句话的九个切面：harness 和权重之间的边界是活的，而且未来几年最有价值的进展，大概率发生在边界上。

如果只看一个数字，去看 METR 的时间跨度曲线；如果只看一个趋势，看记忆和技能怎么从运行时配置变成训练目标。剩下的，让时间验证。
