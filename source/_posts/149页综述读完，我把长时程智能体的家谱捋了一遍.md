上周收到一篇综述的解读邮件，标题是《Towards Long-Horizon Agents: A Survey》，人大、北大、清华、中山、港科大、新国大联合出品，正文 75 页、全文 149 页、引用约 900 篇。这个体量在 agent 综述里算是巨无霸了。

我把 PDF 抓下来自己啃了一遍。这篇先聊它的概念体系（也就是 taxonomy），下一篇再整理里面值得跟的模型、数据集和 benchmark。

## 一句话核心：Agent = π_θ ⊕ H

全文的纲就这一个公式：

> Agent 不是裸模型，而是**基座策略 π_θ 和运行支架 H（harness）的运行时耦合**。

π_θ 是模型权重里已经学到的能力；H 是包在模型外面的那层运行时系统：工作流、上下文、记忆、工具、编排、权限钩子、验证器。⊕ 不是相加，是"耦合"：harness 围绕模型闭合"行动—观察—记忆—再行动"的循环。

这个定义的直接推论很扎心：**评价一个 agent 只说"用了哪个模型"是没意义的**。很多号称的"模型能力提升"，其实来自更好的上下文组织、工具接口或者验证系统。反过来，同一个模型套不同的 scaffold，在 SWE-bench Verified 上的差异，比同一个 scaffold 换不同模型的差异还大。论文引了这个实证，我觉得是全篇最有说服力的一句话。

## 什么算"长时程任务"？不看时长，看依赖结构

作者拒绝用"运行时间"或"token 数量"来定义 long-horizon，而是用**跨越大量相互耦合步骤的逻辑依赖结构**。

举两个例子就明白了：

- 对一百万个文件重复执行同一个操作，跑了好几天，但这是"长运行"，不是长时程，因为步骤之间几乎没有逻辑依赖；
- 一个 30 分钟搞定的复杂软件修复，每一步决策都依赖之前的定位、修改、测试反馈，这就是典型的长时程。

时间只是容易测量的代理指标，horizon 的本质是**决策依赖的跨度**。

## H1 ⊂ H2 ⊂ H3：三级任务，三级能力

论文把长时程难度组织成三个嵌套层级，组织轴是"执行需要超出单个工作上下文多远"：

| 层级 | 任务范围 | 时间量级 | 要求的能力 |
|---|---|---|---|
| **H1** | 单上下文内 | ~分钟 | **C1**：上下文内交互推理 |
| **H2** | 跨上下文、跨会话 | ~小时到天 | **C2**：跨上下文状态与记忆 |
| **H3** | 跨任务、开放式任务流 | ~终身 | **C3**：跨任务经验积累 |

几个我觉得讲得好的点：

- **嵌套关系是严格的能力叠加**。H3 的 agent 在每个任务内部仍然要满足 C1 和 C2。不是升级了就不管基本功。
- 每一级暴露一种新的失败模式：H1 暴露"单步准确率高没用，要的是持续推理"；H2 暴露"上下文窗口再大也没用，还是要压缩、外置、追踪状态"；H3 暴露"单次任务成功没用，要跨任务积累可复用能力"。
- 经验数据上，METR 的时间跨度测量（50% 成功率下能完成的任务的人类等价时长）显示 frontier agent 从 GPT-3 的 9 秒爬到了 Claude Opus 4.6 的 12 小时、Claude Mythos 的 16 小时以上，每 4-7 个月翻一倍，还在加速。按这个趋势，agent 正在从 H1 迈入 H2，开始逼近 H3 的边界。

## 辨析时间：它和那几个兄弟概念到底什么区别

这是全文我最喜欢的一节，因为这些词平时确实被混用得厉害。

**vs. Long-Running（长运行）**：长运行按时长定义，关心持久性和吞吐量；长时程按依赖结构定义，关心耦合推理和状态管理。一个循环跑数据集的脚本跑一周也不长时程；一个单窗口内的密集调试session可以很长时程。

**vs. Autonomous（自主）**：自主是治理维度（人在不在 loop 里），长时程是能力维度（系统能不能在所需跨度上维持推理、状态和经验）。一个强长时程 agent 可以跑在低自主模式（每步审批），一个高自主 agent 也可能只做单窗口短任务。两回事。

**vs. Self-Evolving（自进化）**：自进化要求经验驱动地持久改变自己（权重、记忆、工具、拓扑），是实现 C3 的一种机制。但很多 H1/H2 agent 用固定策略加临时状态就把活干完了，没有任何自进化，它们依然是长时程 agent。自进化关心跨任务持久改进，长时程还关心单次任务内的扩展交互和状态管理。

**vs. Long-Context（长上下文）**：大窗口是必要非充分。模型会系统性地忽略上下文中间段的信息（lost in the middle），利用率过了某个阈值性能还会急剧下降（context rot）。所以 H2 级别的任务，光有 1M 窗口不够，还得有压缩、外置、状态追踪的机制。

一句话总结这组辨析：**长时程问的是"能不能维持一条由大量相互依赖决策组成的完整轨迹"，其他概念各自回答的是"跑多久"、"谁参与"、"会不会变"、"装多少"。**

## 三阶段演进：控制面不断外移

论文把 2020 年至今的历史分成三个嵌套阶段，每一阶段由一次模型能力跃迁开启，由它解决不了的瓶颈终结：

**Stage I：Prompt Engineering（2020–2023）**。GPT-3 的 in-context learning 开启，控制面是**语言**，即怎么措辞。代表技术：CoT、Zero-shot CoT、Self-Consistency、Least-to-Most、Tree-of-Thoughts、ReAct。终结原因：prompt 只能控制语言，给不了模型没见过的东西：实时信息、私有数据、大代码库。

**Stage II：Context Engineering（2023–2025）**。FlashAttention 等高效 kernel 让长序列可用，控制面变成**信息**：每次调用让模型看到什么。代表技术：RAG 全家桶（HyDE、RAPTOR、Self-RAG）、Toolformer/Gorilla 系工具调用、MemGPT、Generative Agents。终结原因：context engineering 本质还是 one-shot 的开放循环，优化的是"模型行动前看到什么"，管不了"行动之后发生什么"。一旦串起多次调用去操作真实环境，失败就变成轨迹级别的属性：误差复合、信念漂移、不可逆副作用。

**Stage III：Runtime Harness（2025–至今）**。可执行环境和 benchmark（WebArena、SWE-bench、OSWorld、Terminal-Bench）提供了轨迹级反馈，控制面变成**整条轨迹**，也就是围绕模型的整个运行时。代表：CodeAct、Reflexion、LATS 这类 loop 的收紧；LangGraph、AutoGen、Magentic-One 这类编排；MCP、AGENTS.md、Agent Skills 这类标准化；以及 Claude Code、OpenHands、SWE-agent 这些产品级 runtime。

注意三个阶段是嵌套不是替代。今天的 agent 依然在做 prompt 和 context engineering，只是控制面的重心移到了轨迹。

## 两根支柱：Harness 六组件 + Optimization 七环节

这是 taxonomy 的主干，论文把现有工作按"能力由外部支架供给还是由训练内化"分成两柱。

**外部化 Harness 六组件**（从控制环向外）：

1. **Loops & Workflows**：调度模型调用、动作、反馈。线性（ReAct）、计划-执行（ReWOO、Arbor）、分支搜索（ToT、LATS）。论文有个很实在的论断：分支搜索只在"坏承诺的成本高于评估候选的成本"时才值得用。
2. **Context & Memory**：把膨胀的轨迹变成有界高信号的工作区。工作上下文三操作：Discard、Compress、Select；持久记忆分事实记忆（AGENTS.md、Mem0、HippoRAG）和经验记忆（Reflexion、Voyager、ReasoningBank）。有个趋势值得注意：从手写压缩规则转向"学习何时/如何折叠状态"。
3. **Tools, MCP & Skills**：工具接口协议化（MCP）、工具主动发现（检索式注入、CodeAct 程序化调用）、技能库作为一等可演化资产（Voyager、Agent Skills、SkillNet）。论文特意强调技能和 MCP 是正交的：MCP 给连接性，技能给程序性知识。
4. **Orchestration**：分解与角色、协调拓扑、拓扑优化、agent 协议（A2A、ACP）。关键实证：多智能体失败大多怪角色欠规范、agent 间错位、缺验证，而不是基模型弱；"角色人设"的收益其实来自它诱导的结构分解，不是人设本身。DAG 是主导抽象。
5. **Hooks & Middleware**：动作路径上的拦截器。预定义规则（输入/工具调用/协议三边界）、用户自定义（符号引擎/LLM 编译/LLM 运行时评估）、运行时自适应（不确定性驱动/行为驱动/威胁驱动）。
6. **Verification**：评估引擎，正在从"跑完打分"走进轨迹内部：步级 PRM、领域专用 PRM（Web-Shepherd、SWE-PRM）、到自己会调用工具探查环境的 agentic verifier。核心论断是 tool-grounded 的外部验证远比模型自我反思可靠。

**内部化 Optimization 七环节**（沿训练管线）：

架构基底（显式上下文 / 压缩状态 / 混合 / 高吞吐）→ 数据与环境合成（任务、环境、轨迹三条线）→ 预训练与中训练（推理先验、长上下文状态、多模态感知、数据配方）→ 微调（指令选择、课程学习、轨迹蒸馏）→ Agentic RL（信用分配、策略优化、采样策略、交互模式）→ On-Policy 蒸馏（教师引导、自我改进）→ 自进化（离线、在线、agent-环境共进化）。

两柱之间的关系不是竞争而是**迁移**：harness 里先跑通的机制，会被逐步蒸馏进权重；更强的权重又解锁更复杂的 harness。Prompt engineering 时代的 instruction tuning 是第一次迁移，现在的 Search-R1、SWE-RL 这类 agentic RL 是正在发生的迁移。

## 失败模式清单

论文归纳的几类典型死法，做工程的对照一下基本都有体感：

- **Goal drift + 误差累积**：单步成功率 p，n 步全对就是 p^n，局部错误还会改变后续状态分布；
- **Context rot**：上下文填充过阈值后性能急剧下降；
- **Context-limit 压力**：agent 感知到上下文快满了会提前收工，把半成品说成完成；
- **奖励稀疏延迟**：很多任务只在结束时有信号，中间决策没有学习信号；
- **不可逆行动**：跑得越久，越可能干出收不回来的操作；
- **经验复用污染**：agent 从自己的经验里蒸馏出危险启发式并复用（观察到"退款带来好评"→ 过度泛化为倾向退款），而且会随着技能库、共享记忆跨团队传播。

## 我的几点辨析

夸完了，说几点保留意见：

第一，**harness 这个概念被用得极宽**。从 system prompt 到 hooks 到验证器全算 harness，宽到有点"凡是权重之外皆 harness"的意思。好处是统一了话语，坏处是做归因分析时这个筐太大，一篇论文报告的提升到底来自六组件里的哪个，仍然需要逐案消融。

第二，**H1/H2/H3 的边界靠"上下文窗口"这个操作代理来划，但窗口本身是变动的**。1M 窗口普及后，很多昨天的 H2 变成了今天的 H1。这个 taxonomy 的时间稳定性存疑，论文自己也承认层级划分依赖操作性代理指标。

第三，**三阶段编年史有点过于整齐**。真实历史里 ReAct（2022）已经有 loop 了，Voyager（2023）已经有技能库了，按论文分期它们属于"Context Engineering 时代"，但它们的控制面明明已经是轨迹级的。阶段划分适合做心智地图，不适合做严格的编年。

第四，**"时间跨度每 4-7 个月翻倍"这类数字要谨慎**。它来自 METR 特定任务套件上的 50% 成功率拟合，方法论换一版 doubling time 就从 6.5 个月变 4.3 个月。趋势是真的，具体斜率别太当真。

但这些都不妨碍它成为目前这个领域最完整的一张地图。对于"该把精力投在模型侧还是 harness 侧"这个实际问题，论文给出的答案很明确：**下一阶段的进步越来越是运行时的属性，harness 而不是模型，是生产环境 agent 的主要 alignment 和能力 surface**。我基本同意。
