这是《Towards Long-Horizon Agents》系列笔记的第三篇。第一篇[拆了术语](https://dxyang42.github.io/2026/07/19/149页综述读完，我把长时程智能体的术语挨个拆了一遍/)，核心是 Agent = π ⊕ H，模型加 harness 才是完整的 agent。第二篇[整理了资源清单](https://dxyang42.github.io/2026/07/19/长时程智能体资源清单：哪些模型、benchmark、数据集值得盯/)。这篇钻进 π 内部：模型侧到底怎么优化，长时程能力是怎么一点点长进权重里的。

综述 §5 的标题就点明了主线：Internalizing Long-Horizon Capability，内化。harness 是运行时外挂，模型侧优化的本质是把外挂干过的活收进权重。原文说得很直白：流水线每一阶段要么在内化一个原本由运行时提供的能力，要么在为下一步内化生产经验。

整节组织成一条七级训练流水线，我按这个顺序拆。

| 阶段 | 干什么 | 内化的对象 |
|---|---|---|
| 5.1 架构底座 | 决定历史怎么存 | 长序列表征能力 |
| 5.2 数据与环境合成 | 造任务、造环境、造轨迹 | 训练经验的生产线 |
| 5.3 预训练/中训练 | 埋先验 | 推理、长上下文、多模态 |
| 5.4 SFT | 过程级模仿 | 交互模式、工具用法 |
| 5.5 Agentic RL | 从自己交互里学 | 延迟奖励下的决策 |
| 5.6 On-Policy 蒸馏 | 在自己的轨迹上被教 | 分布偏移下的纠错 |
| 5.7 训练时自进化 | 数据任务环境全进循环 | 经验写回权重的闭环 |

## 5.1 架构底座：历史怎么存

模型要怎么记住一条超长轨迹，架构层面有四种回答。

| 路线 | 思路 | 代表 | 短板 |
|---|---|---|---|
| 显式上下文 | 历史就是 token 序列，靠 attention 互查 | Longformer、Big Bird、NSA、MoBA | 成本和窗口一起涨，稀疏模式会漏关键 token |
| 压缩状态 | 历史压成紧凑内部状态，逐步更新 | RetNet、RWKV、Mamba | 精确回忆不行，旧 token 折进状态就找不回原文 |
| 混合架构 | attention 管精确取证，状态模块管长程连续性 | Jamba、Kimi Linear | 工程复杂度高 |
| 高吞吐机制 | 不改存储方式，只降解码的串行开销和缓存流量 | 投机解码、Medusa、EAGLE、MTP、MQA/GQA | 省的是钱，不解决记忆问题 |

两个细节值得单说。

一是 MLA 的归类。DeepSeek-V2 提出、V3 沿用、Kimi K2 也在用的 MLA，把 KV cache 压成 latent。看着像压缩状态，但它保留了每个 token 的 latent，依然对全部历史做 attention，所以没有精确回忆的损失。综述把它定在显式上下文和压缩状态之间，兼作高吞吐机制。这个辨析挺能说明问题：架构分类不是贴标签，要看信息到底以什么形式被保留。

二是 MoE 为什么不在表里。综述明确说 MoE 不算第四种表征路线。它路由的是 FFN 专家，改变的是模型容量，不决定序列历史怎么存。所以 MoE 模型照样可以配显式 attention 或混合架构，两个维度别混在一起。

对长时程 agent 来说，这个选择的实际含义是：纯 attention 能精确翻旧账但贵，纯压缩状态便宜但会丢细节，而 agent 的活恰恰既要翻旧账（三步前的工具返回原文）又要长跑。混合架构成为方向不是偶然。

## 5.2 数据与环境合成：没有经验就造经验

长时程行为和普通问答最大的区别，是它学的是交互经验：怎么决策、怎么观察反馈、怎么纠错、怎么在长轨迹里攒信息。这种数据天然稀缺，部署日志贵且有隐私问题，所以合成成了主路线。三个耦合的对象：任务、环境、轨迹。

**任务合成**四条路：

| 路线 | 思路 | 代表 |
|---|---|---|
| 依赖链 | 把子目标串起来，早期决策约束后期动作 | AgentSynth、TaskCraft |
| 图结构信息 | 任务是从网页/文档/证据图里找路径，难在分支和不完整观察下选证据 | WebWalker、WebDancer、WebSailor、WebShaper |
| 状态转移 | 任务=初态到目标态的差，或从错误态恢复 | SWE-Gym、R2E、SWE-smith、DIVE、CLI-Gym |
| 课程驱动 | 难度跟着当前策略走，不提前定死分布 | WebRL、AgentFrontier（最近发展区）、CuES |

前三条决定"考什么"，课程驱动决定"难度涨多快"，两个维度正交。

**环境合成**三族：

| 族 | 转移和反馈谁产生 | 代表 | 问题 |
|---|---|---|---|
| 符号环境 | 代码、测试、数据库、业务逻辑，硬信号可验证 | SWE-Gym、Terminal-Bench、WebArena、OSWorld | 规则明确但扛不住模糊场景和动态界面 |
| 神经环境 | 模型当世界模型，直接模拟下一轮反馈 | DIAMOND、GameNGen（像素级）、WebDreamer（文本级） | 便宜可扩展，但会幻觉、会漂移、有 sim-to-real 差距 |
| 混合环境 | 硬状态和最终验证留在符号侧，模型管用户模拟和语言接口 | τ-bench、MCP 工具服务器 | 设计核心是模型和符号验证的边界画在哪 |

**轨迹合成**四个方向：环境接地的执行记录（TOUCAN 在真实 MCP 环境里合成、ASTRA 用工具调用图约束）、策略对齐的 rollout（AgentGym-RL、Agent Lightning 把执行记录拆成 RL 可消费的单位）、步骤级归因和失败复用（CSO 找改变结果的关键步、AgentHER 给失败轨迹换个可达目标重新标注）、历史压缩（ReSum 按摘要分段训练、Context-Folding 把子过程折叠）。

这块有个判断我觉得很关键：轨迹质量继承环境的质量。环境能观察什么、能验证什么，决定了轨迹里能留下什么信号。垃圾环境造不出金轨迹。

## 5.3 预训练与中训练：先验怎么埋

后面 SFT 和 RL 能激发出的能力，上限在预训练就定了个大概。四个抓手。

**推理先验**。代码和数学是注入推理先验最好的数据源，因为它们天然带结构化依赖、可执行程序、多步推导和可验证的中间状态。Qwen2.5、DeepSeek-V3、GLM-4.5/5、Kimi K2 的配方里，代码数学推理和 agentic 数据都是重头。Qwen3-Coder-Next 更直接，中训练就瞄准代码推理、仓库级理解和 agent 式交互。

**长上下文状态**。对 agent 来说，上下文不是更长的输入窗口，是序列化的工作状态，里面装着目标、观察、证据、工具输出、失败记录和改过的计划。两条技术线：一条改位置编码，Position Interpolation 线性缩放、YaRN 按频率分档、LongLoRA 用移位稀疏注意力做参数高效的扩展；另一条改训练配方，ProLong 证明长上下文能力取决于继续训练的数据构成和序列长度，UltraLong 推到百万 token，LongWriter 则发现光有大窗口不会自动带来长输出，得专门训。GLM-5.2 把 1M 窗口和长时程编码场景的专项训练绑在一起，算是这个思路的工业落地。

**多模态感知**。agent 面对的任务状态分布在网页、截图、视频、音频流里。趋势是视觉接地和 GUI 控制从预训练阶段就当原生能力训，而不是文本模型训完再补。Seed1.5/1.6-VL 用大规模图文和交互数据交织预训练，MiMo-VL 把 GUI 定位当一等目标，Qwen3.5-Omni 和 Kimi K2.5 往全模态走。

**数据配比**。DataComp-LM、FineWeb、Dolma、OLMo 让预训练数据构建可复现；DoReMi 和 Data Mixing Laws 用代理实验优化领域配比；OPUS 在每个训练迭代里按样本在优化空间的效用打分选数据，从"更多 token"转向"更好的 token"。对 agent 更直接的是把数据组织成轨迹而不是孤立文本，AgentTuning、Agent-FLAN、AgentBank 都验证了这个方向。

## 5.4 SFT：过程级模仿

长时程 agent 的 SFT 不是指令跟随，是对轨迹的过程级模仿：怎么分解目标、怎么调工具、怎么读观察、怎么更新状态、怎么从错误里爬出来、怎么收尾。它通常当冷启动，先把合法交互模式框住，再上在线优化。

**选数据，质比量重要，但"少即是多"有争议**。AgentInstruct 用一小撮高质量轨迹混通用指令；FireAct 证明强模型生成的轨迹比纯 prompt 适配管用；LIMI 主张策略性精选的小样本就能激发 agency。更激进的 ATLaS 干脆不克隆完整轨迹，只挑规划、子目标推理这些关键步骤微调，反而减轻了专家偏置、提升了跨环境泛化。LIMO 用 817 条精选样本从强基座里引出复杂数学推理，s1 压到 1K。反对声音也有：Ding 等人的实验显示，SFT 接 RL 的流水线里，数据规模仍然决定性能天花板，难度只是乘数。我的读法是，小样本激发的是基座里已有的东西，要突破上限还是得堆覆盖度。

**工具数据得可执行**。CodeAct 把动作表示成可运行的 Python，带执行反馈和自我调试；APIGen 的函数调用数据过格式、执行、语义三重校验；ToolACE 用自演化 API 池扩产；各家数据格式太碎，Agent Data Protocol 出面统一成公共轨迹表示，跨域混合 SFT 收益明显，还能避开单域微调的负迁移。

**课程学习按时程和难度两个维度排**。AgentGym-RL 的 ScalingInter-RL 先用短时程压榨出稳定基础策略，再逐步放大交互预算，躲开了直接上长任务的优化崩塌。Kim 等人的对照实验更干净：只改动作序列长度这一个变量，就证明时程本身是训练瓶颈，而且先短后长能诱导时程泛化，短任务上学到的能力能迁移到没见过的长任务。

**蒸馏是轨迹蒸馏，不是答案蒸馏**。强到弱方向，Agent Distillation 把检索和代码执行轨迹蒸进 3B 以下的小模型；Structured Agent Distillation 把轨迹切成 [reason] 和 [act] 段分别算损失；AgentArk 把一整个多智能体辩论系统蒸进单模型，把昂贵的测试时交互变成隐式自纠错。自蒸馏方向我最喜欢 Agent-R：用 MCTS 在失败轨迹里定位第一个错步，把它剪接到相邻的正确路径上，迭代微调教的是"当场改"，不是"跑完整局再忏悔"。RESD 则把失败转成复盘诊断，攒成可复用的 playbook。

## 5.5 Agentic RL：四个问题

SFT 学的是别人的轨迹，RL 优化的是自己的交互。综述把 Agentic RL 拆成四个问题，这个拆法比一堆方法名好记。

### 问题一：credit 怎么分

| 粒度 | 思路 | 代表 | 毛病 |
|---|---|---|---|
| Outcome | 只看最终结果：成功率、单测、终态检查 | DeepSeekMath、Search-R1、DeepRetrieval | 一个终分摊给整条轨迹，关键探索和冗余动作分不清 |
| Process | 给中间决策打分，或把稠密奖励铺到过程里 | ToolRL（把工具调用拆成格式、选型、参数几个维度分别打分）、AgentPRM | 需要过程监督源 |
| Rubric | 单指标不够时，用显式多维语义标准 | RaR、OpenRubrics（从偏好对比里归纳规则）、DR Tulu（rubric 和策略共演化）、RuscaRL（rubric 当会衰减的探索脚手架） | rubric 质量、judge 偏置、reward hacking |

rubric 这条线是 2025 年下半年以来的热点，原因很实际：agent 任务的上下文依赖、工具副作用太容易被单一复合奖励掩盖掉。

### 问题二：策略怎么更新稳

长轨迹加稀疏二元奖励，PPO/GRPO 家族的毛病全出来了：长度偏置、组内归一化偏置、全对全错的无效组、token 级重要性比率不稳。clipping 系打补丁：REINFORCE++、DAPO、Dr.GRPO 修统计，GSPO 把重要性比率和裁剪提到序列级。

更有意思的是优化单元下移。决策单位本来就不是单次响应，而是 turn、step、子目标。Turn-PPO、StepPO、SegPO 把 advantage 对齐到显式分段；GiGPO、Tree-GRPO 把比较锚定到共享状态或兄弟分支上，回答"哪个局部选择改变了后续轨迹"。

第三个麻烦是熵坍缩：长时程 RL 很容易过度强化早期成功模式，轨迹多样性迅速消失。Clip-Cov、KL-Cov 给过度集中的更新装闸门；CURE 围绕高熵关键 token 重新拼接 rollout；EPO 把熵正则提到轨迹级；ARPO、AEPO 在工具反馈后的高熵位置做分支采样。

### 问题三：轨迹怎么采

采样决定训练看到什么。轨迹级 rollout 是基本单位，Search-R1 一系把查询生成、证据获取、停止决策全折进 on-policy 轨迹，WebDancer、WebSailor 家族建了端到端的深研采样流水线。树结构采样解决独立 rollout 贵且对比信号弱的问题：从 ToT、RAP、LATS 一路到 Tree-GRPO，共享前缀、兄弟分支互相对比，TreePO 还能复用推理算力。预算感知剪枝管成本：LiteResearcher 按通过率剪，TCOD 用轨迹深度课程，ARPO 只在高熵工具调用步触发分支。

### 问题四：交互结构怎么组织

三个方向。层级框架：GLIDER、STEP-HRL 做规划器-执行器分解，子目标边界变成新的优化单元；SkillRL 从 RL 失败里演化技能库，Skill0 则直接把技能内化进参数。多智能体：MAGRPO 集中式 advantage 加分散执行，M-GRPO 给规划器和工具执行子 agent 分别算组相对优势，Dr.MAMR 用 Shapley 式因果影响估计治"懒 agent 坍缩"。

最值得说的是**记忆状态学习**：把本来由 harness 固定管理的上下文、工作流、长期记忆，直接变成可学习的策略。Memory-R1 拆出独立的 Memory Manager 和 Answer Agent；Memory-as-Action 把工作记忆编辑当显式策略动作；Agent Workflow Memory 从成功任务里归纳可复用工作流；Agent Lightning 把现有工作流 trace 成 span 和转移，转成可训练样本。这一支是 π 和 H 融合的最前线：harness 的组件正在被一个一个变成 RL 的动作空间。

## 5.6 On-Policy 蒸馏：中间路线

离线蒸馏吃固定专家轨迹，学不到自己走出来的状态分布；RL 在轨但奖励稀疏、探索贵。On-Policy Distillation（OPD，GKD 最早形式化）走中间：学生跑自己的轨迹，教师在这些轨迹上给监督。综述特意划了界：和 5.4 的离线蒸馏比，OPD 的监督落在学生当前 rollout 上；和 5.7 的自进化比，OPD 的任务集和环境固定，只是把监督信号加密。

教师引导一系：DAgger-LLM 做 turn 级插值和逐步监督；MAD-OPD 聚合多个教师；KAT-Coder-V2 把多个互补专家 agent 蒸进统一策略；LiteGUI 用检索到的参考轨迹增强监督。核心发现是监督质量不只取决于教师强不强，还取决于什么时候、以什么粒度注入。

自我提升一系不要外部教师，用两类自监督信号。一类是特权信息：训练时可见、部署时不可见的东西，比如结果信息、细粒度视觉标注（π-Play 是代表）。另一类是环境反馈：把稀疏的奖励、报错、验证器输出转成稠密自监督，或者把失败转成反思攒下来。

## 5.7 训练时自进化：闭环

前面所有阶段都消耗预先准备好的数据和任务。自进化把反馈环闭上：当前策略产生新尝试、新失败、新反馈，训练把筛过的记录写回模型。数据、任务、甚至环境都变成循环自己更新的对象。三种形态：

| 形态 | 循环怎么转 | 代表 | 风险 |
|---|---|---|---|
| 离线自进化 | 生成-筛选-训练，数据可审计可降噪 | STaR、SOAR、SAMULE、rStar-Math（MCTS 合成逐步验证的轨迹，策略和过程偏好模型共同演化） | 候选数据来自现有分布，强化已知多、发现新策略少 |
| 在线自进化 | 经验采集和策略更新交替，在自己诱导的状态分布上学 | WebRL、Agent0（课程 agent 和执行器从同一基座共进化，零外部数据） | 对环境和验证器质量极敏感，长 rollout 放大一切 |
| Agent-环境共进化 | 环境本身跟着策略能力调 | Environment Tuning、RLVE、GenEnv、Agent-World、Socratic-Zero（Teacher/Solver/Generator 三方共进化） | 动态课程可能强化有缺陷的能力 |

综述给的判定标准很严：外部记忆、技能库、工作流痕迹只是经验的中间形态，只有当它们的稳定模式通过蒸馏、微调、RL 或可训练模块被真正吸收，才算训练时自进化。风险也来自 bootstrap 结构本身：自生成数据可能让分布收缩，弱验证器诱导 reward hacking。所以结论是需要强验证器、过程级反馈、离线审计加在线探索的组合。

## 收尾：我的几点看法

第一，这节的组织方式本身就值得学。大部分综述按能力列方法，这篇按训练流水线组织，每一级回答"这一棒把什么内化进权重"，和 §4 的 harness 六组件正好镜像。读完之后看任何一篇 agent 训练论文，都能定位它在流水线的哪一级。

第二，内化是个大方向，但不是免费的。运行时外挂改起来是改配置，内化进权重改起来是重新训练。harness 的价值不会消失，它只是不停把沉淀下来的能力交给权重，自己往更前沿的地方挪。

第三，如果让我押一个这节里最值得跟踪的方向，我押记忆状态学习。它直接回答了第一篇里 C3 能力（经验的经验性）怎么来的问题，而且是七级流水线里唯一一个同时横跨架构、RL 和自进化三段的方向。
